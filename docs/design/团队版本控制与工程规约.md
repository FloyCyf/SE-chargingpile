# 团队迭代与大盘源码防冲突管理法则

无论您的小组是通过 Gitee、GitHub 进行极其纯正的前沿协同代码分支演进，亦或者哪怕是以微信互通局域网的发送压缩文件包为转移锚点。
本项目的源基建被强制规定应遵循这套“提交命名+代码底线”的规约来确立每一笔交付记录的历史。这也是面向指导教师证明你作为现代开发者**基本素质与协作觉悟**的最好军规！

## 第一条铁手腕：源码风格的一体化同频

既然是合作项目，就绝不允许出现“甲写的风格是这样，乙写的又是散装一地”的拼接感。

* **禁止徒手写死配置参数**：任何涉及到在代码中赋值的算法权重 $\alpha$ 参数、惩罚上限等调度常量，一律强制收回到根目录统一管制的 `config.yaml` 字典内；绝对不允许污染具体的业务文件。
* **原生 Types 解析化注解**：所有 Python 代码层，凡是非基础且不易被大脑编译器读出业务特性的重要参数及返回值类型，强行要求追加 `: Type` 或者是 `-> type` 返回值提醒。以消除后续交接计费引擎或测试打回时的联防痛点。

---

## 第二条生命线： Conventional Commits (标准格式提交)

任何时刻要封版、提交、或是打包为新的合并代码点移交给你的组长或下一个接手开发的协作者之前，你的那条用于记录“这段时间干了点啥”的消息，被严格界定为必须遵循如下的前缀标枪：

你提交代码的名字，或者命名这个压缩包版本的名字前缀（强制二选一要求），仅允许从这四大核心标准标签中认领：

* `feat(组件): XXX`：用于你为系统新写完了一套全新的子计费模块、加装了一个完全没有的算法函数或者是开辟了针对性的业务时使用。
  > ✓ 正确例： `feat(Scheduler): 在队列调度里补装熔断机制以防御爆满拦截`
* `fix(文件): XXX`：由于跑偏出现的参数失控、数据库没锁上跑断了的问题在您手中重写排雷后使用。它标注了此次交卷是为了终结某个 Bug。
  > ✓ 正确例： `fix(Admin): 修正 Admin前端当遇到断网轮询时没有消灭弹窗警告的逻辑毛病`
* `docs: XXX`：如你更新了架构设计与需求规格说明书等软性文件，用这个前签。
* `style(HTML|Python): XXX`：专门用作将原生面板改的更加赛博、替换了不规范的空格为标准的 Pep8 等强视觉改变的情况。

## 第三条纪元纲领： 支线任务切割
在团队开发任何跨端逻辑（比如：即将要去实施难度极高的基于数学权重的优先级洗牌更新）时，“绝对不允许”几个人在同一套完整的底座上相互覆盖开发。

必须按人头或者增量版本为基础切分出去一个功能试验版（即 Pull Request 所依托的发展分支 / 分散小包）。待这个单点算法在本地跑出了完美无冲突的 `VirtualClock` 沙盘推演结果后，才能合并覆盖大家共享的核心 `Main`（总应用系统干线）。从而从根本物理机制上杜绝由于一个代码的半成品带入核心池导致整个团队框架奔溃的系统性悲剧惨烈发生。
