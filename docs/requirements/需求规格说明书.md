
# 智能充电桩调度计费系统 - 软件需求规格说明书 (SRS)

## 1. 引言
### 1.1 目的
本文档旨在定义波普特大学（BUPT）智能充电桩调度计费系统的功能、约束与性能要求，为增量式开发提供统一的技术准则。

### 1.2 项目背景
针对校内新能源车主日益增长的充电需求，系统需通过智能调度最小化用户的等待与充电总时长，并实现透明的计费管理。

---

## 2. 系统约束与环境
### 2.1 硬件配置参数（配置化）
为保证验收时的灵活性，以下参数应作为**系统配置参数**（如通过 `config.yaml` 或环境变量加载），不应硬编码在代码中：
* **充电桩资源**：系统支持“快充桩”与“慢充桩”，每台桩仅支持单一模式。
* **资源数量**：快充桩数量 $N_{fast}$、慢充桩数量 $N_{slow}$ 需动态可调。
* **物理空间限制**：等候区（车位）容量 $M$。
* **状态定义**：单个桩包含“正在充电”及“正在排队”两个状态位。

### 2.2 运行环境与技术选型
为了追求无跨域困扰、零门槛装配的极速本地演示体验，本系统拒绝分离式编译环境，必须实行前后端在同一端口一体化托管，规范严格如下：
* **稳定底层后端**：Python 3.9+ (搭载现代化的异步 FastAPI 框架体系构建)。
* **原生极简前端驱动**：**强约束禁止**在项目中引入 Vue / React 等重型的前端工程化套件和 node_modules。系统中包含的车主端与中控台界面需且仅需采用最轻量的 **纯正原生 HTML5 骨架 +原生 Vanilla Javascript (配合内置 Fetch API 进行双向通讯) + 原生 CSS 层叠样式**。
* **交付大屏形态**：
  - **车主高频触点 (客户端 UI)**：原生渲染的轻量网页。车主能在此发送打车般顺滑的请求表单，通过返回面板扫视等位、预测充电所需时间等系统级响应。
  - **站内数据枢纽 (Admin 中控)**：长轮询挂载的数字罗盘展示页。供给站场保安与管理员实时巡查 6 大（或更多）充电子接口的在桩率，同时呈现未来预装的图表和单据流水。

## 4. 功能需求细化

### 4.1 账号与车辆管理
* **统一身份认证**：支持校内身份登录，记录用户信誉分（与违约惩罚挂钩）。
* **请求信息录入**：用户提交请求时需明确：
  1. 车辆 ID (车牌号)
  2. 充电类型 (快充 Fast / 慢充 Slow)
  3. 当前电量 (SOC, 0.0-1.0)
  4. 目标充电量 / 目标 SOC
* **用户主动操作与终止约束（完善项）**：
  1. **无责取消**：允许车主在尚未分配到充电桩（处于等候队列状态时）撤销请求，系统不收取任何费用并在队列中剔除该用户。
  2. **主动提前断电**：允许车主在充电期间（未充到设定目标 SOC）主动终止并拔出充电子，系统需触发结束调度事件，并立即按当前真实充入的电量及时段落闸，生成最终账单。

### 4.2 智能调度模块（核心）
系统采用**动态优先级调度算法**，而非简单的先到先得。

#### 4.2.1 优先级评分公式
对于排队队列中的任一车辆 $i$，其优先级评分 $P$ 计算如下：
$$P_i = \alpha \cdot (1 - SOC_i) + \beta \cdot W_{type} + \gamma \cdot \ln(t_{wait\_i} + 1)$$
* **$SOC_i$**：当前剩余电量。
* **$W_{type}$**：快充请求权重取 1.0，慢充取 0.5。
* **$t_{wait\_i}$**：在该请求类型队列中的累计等待时间。
> **【模型参数约定】**：公式内决定的参数 $\alpha$（电量饥渴权重）、$\beta$（模式权重）、$\gamma$（等待时间权重）**必须禁止在代码中硬编码写死**，需要与车辆等候容量一样，放置在配置文件 `config.yaml` 或管理员参数后台中支持实时热更新，用以应对未来负载调优。
* **调度逻辑**：当对应类型的充电桩空闲时，系统从队列中选取 $P$ 值最高的车辆分配资源。

#### 4.2.2 队列管理逻辑
* **双队列设计**：快充与慢充维护各自独立的优先级队列。
* **容量控制**：当“正在充电人数 + 队列排队人数 $\ge$ 等候区容量 $M$”时，系统应拒绝新请求。
* **锁定机制**：分配桩后，系统预留 15 分钟入场缓冲。若未感应到“插枪”，则自动释放并惩罚该车主。

#### 4.2.3 调度触发条件

系统按照“事件驱动为主，时间推演为辅”的原则触发优先级重新计算及分配。明确规定以下 **5种场景** 产生时，系统必然被唤醒并介入调度更迭：
1. **用户主动提交新请求（入队事件）**：当新车主呼叫充电且系统判定同类桩尚有空闲时，不一定直接分配，须对全队列优先级权重排序后，分配给最高权重车辆。若无空闲桩则仅追加至队列末尾。
2. **充电达成目标自动释放（完成事件）**：当有车辆充满或达到设定的目标 SOC，供电系统自动落闸并释放该桩，物理坑位空出，触发调度并将最高优先级等候用户调入。
3. **用户主动提前中断释放（中止事件）**：处于充电中的用户因特殊原因提前拔出充电子，任务未达原定 SOC 即被迫结束，系统需立即回收空闲桩并开展新一轮调度分配。
4. **充电桩故障或检修掉线（异常事件）**：当工作中的充电桩突然发生故障离线时，系统必须强制断开该桩，并将原在此桩充电的车辆挂起结算，同时触发调度机制利用其他空闲桩（如有此等情况）。
5. **系统时钟定期刷新推演（时间推演事件）**：即便整个场站静默、无新增请求且无车拔枪，因队列中等待者的已随等待时长（$t_{wait\_i}$）随着时钟推移增加，可能导致优先级得分发生“反超排位”，影响顺位情况。

### 4.3 计费与监控模块
* **实时监控**：前端需展示每个桩的编号、类型、当前车辆、实时电量及预计完成时间。
* **费用定价标准（参考基准）**：
  为方便计费模块的开发统一核准标准，系统初始采用以下常数设计：
  - **基础充电服务费**：全天固定收取 `0.80 元/度`。
  - **分时阶梯电费（仅度数基础费）**：
    - **【波峰时段】** `10:00-15:00` 及 `18:00-21:00` —— 费率：`1.00 元/度`
    - **【波平时段】** `07:00-10:00`、`15:00-18:00` 及 `21:00-23:00` —— 费率：`0.70 元/度`
    - **【波谷时段】** `23:00-次日07:00` —— 费率：`0.40 元/度`
  - **超期占位惩罚**：给满电车主单次操作的豁免期为 30 分钟，逾期后，按照 `1.00 元/分钟` 极速叠加收费以驱逐占用了公共资源的车辆。
* **计费策略与费用栈**：
  充电账单总体由三部分累加得出：**最终账单金额 = 阶梯基础电费 + 基础服务费 + 超时占用费**。
  1. **分时区间电价与服务费**：依据真实的充电时长与度数，将其按起止时间切分出所落的天然区间（波峰、平、谷），各时段独立乘算加和。
  2. **超时占用逆向惩罚**：电量充满（或充至目标值）结算落闸后，逾期未按时拔出插枪的车辆，按上列滞纳金单价和超期分钟数直接累加扣费。

* **详单与最终账单必备数据字段标准**：
  每次完成服务落地存入数据库及发送用户的详单中，必须且最少提供以下要素：
  - **单据凭证号**：全局唯一流水账号 (`order_id`)。
  - **实体溯源**：绑定服务的车牌号 (`vehicle_id`)、使用的充电桩组及编号 (`pile_id` & `charge_type`)。
  - **时间溯源线**：自提交排队入站系统起算的时间、确切的插枪供电起算时间 `started_at`、计费落闸（断电或主动）时间 `finished_at`。
  - **耗电详情**：入场表显电量 `start_soc`、结账表显电量 `end_soc`，与经系统核查确认注入的总比例。
  - **收费明细项**：切片详参出本结算期的【**阶梯电费花销**】、【**充电附加服务费**】、【**超时惩罚产生的补偿金**】，以及上述项综合核准出的【**最终应结款支付总额（`final_total_amount`）**】。

### 4.4 消息通知
* 包含“调度成功、开始充电、充电异常、充电结束、产生欠费”五个触发点的实时推送。

## 5. 性能与非功能需求
* **响应性**：调度算法对 $N=100$ 级别的计算需在 100ms 内完成。
* **可伸缩性**：系统数据结构设计需支持后期增加“中充”或“V2G（反向供电）”等新模式的扩展。
* **数据持久化**：所有订单及充电记录需异步写入 SQLite/JSON 数据库，确保断电数据不丢失。

## 6. 增量开发阶段划分与前端要求
为保证与客户或验收评委能够进行直观、无障碍的阶段性交付验证，系统的每个增量阶段都必须采用**“底层数据逻辑 + 可视化前端界面”配对产出**的模式开展：

1.  **增量 I (核心框架与初期双端交互界面)**
    * **后端研发**：建立 FastAPI 架构、接入 SQLite 数据库连通、落地基础的 FIFO（先到先得）调度器，及构建虚拟相对时钟的后台心跳运行机制。
    * **前端配套**：**在第一阶段即提前双开并搭建【简易客户端交互看板】与【基础 Admin 管理员数据大屏】**。彻底告别黑框框，支持客户和评委在页面提交车辆进场发请求，并能在管理员看板实时看到充电桩从“空闲”变“忙碌”的简单映射以及直观的出列动效。
2.  **增量 II (智能优化与全功能车主端升级)**
    * **后端研发**：彻底抛弃先到先得模型；基于电量饥渴度、模式与等待时间的数学公式集成动态优先级计算核心算法，并完善快慢桩的完全隔离解耦调度。
    * **前端配套**：基于第一版的骨架**精细化升级车主客户端展示层**。支持渲染每位车主实时的以分为刻度的充电进度（SOC）爬升组件、达到满电的细致剩余倒计时、以及动态展现“由于数学权重逆转而使得某台车强行插队”的炫酷反馈机制。
3.  **增量 III (计费核算与运维 Admin 总枢纽完善)**
    * **后端研发**：封装完成波峰平谷分时阶梯计费引擎、三十秒级逾期滞纳金收缴扣罚系统、与历史账单电子快照查询器。
    * **前端配套**：基于第一版的版图**深挖拓宽 Admin 大屏业务线**。挂载现代化的流水走势曲线图展示收费站营收，加载发票与详单电子留底查询格窗，和供管理员热更调度算法中权重系数的管理表单面板。
