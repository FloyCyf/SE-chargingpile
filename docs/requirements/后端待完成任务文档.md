# 后端待完成任务文档

本文档基于《智能充电桩调度计费系统详细需求 20260327(1).docx》、当前代码实现和前期检查结论整理，面向后端开发人员。目标是把系统从“能演示基本流程”推进到“满足详细需求、可稳定验收”的状态。

## 一、当前后端状态

已具备的能力：

- 使用 FastAPI 提供用户、管理员、兼容前端三类接口。
- 已有 `users`、`vehicles`、`charge_orders`、`charging_piles`、`pile_queues`、`pile_status_logs`、`bills`、`bill_details` 等模型。
- 已从 Git 跟踪中移除 `charging.db`，并加入 `.gitignore`，数据库作为本地运行产物处理。
- 服务启动时会创建数据库表，并按 `config.yaml` 种入 F1/F2/F3/T1/T2 充电桩。
- 调度器已有快充/慢充队列、最短完成时间分配、虚拟时间推进、计费、账单/详单生成、故障/恢复处理雏形。
- 管理端已有账单、详单、订单 CSV 导出接口。

主要风险：

- 后端接口存在多套入口：`/api/user/*`、`/api/*`、`/api/admin/*`、兼容接口 `/api/vehicle/*`、`/api/user/charge/*`，需要统一契约。
- 调度运行状态主要存在内存中，数据库中的 `charging_piles`、`pile_queues` 尚未完整反映实时队列。
- 普通调度当前是“等候区同类型 FIFO 叫号 + 选择最短完成时间充电桩”，这与本次详细需求和验收 Excel 的主调度策略基本一致，但需要补齐测试闭环、稳定规则和验收快照输出。
- 虚拟时间只支持倍率推进，不支持按验收表固定从 06:00 开始，也不支持按 Excel 时刻推进，跨峰/平/谷演示仍不够可控。
- 每桩队列长度 M 的语义需要统一：验收表要求每桩 3 个车位，即 1 个正在充电 + 2 个排队。
- 故障重调度需要明确本次验收默认采用 A 策略：故障队列优先，期间暂停普通等候区叫号。
- 动态优先级评分策略不作为本次验收 P0 主线，调整为扩展需求或后续优化项。

### BE-P0-04 固化本次验收主调度策略

需求依据：

- 《智能充电桩调度计费系统详细需求 20260327(1).docx》要求：
  - 快充请求生成 F 类型排队号，慢充请求生成 T 类型排队号。
  - 等候区按充电类型分别排队。
  - 当同类型充电桩队列存在空位时，选取等候区中该类型的第一辆车进入充电区。
  - 被调度车辆进入充电区后，选择“等待时间 + 自己充电时间”最短的同类型充电桩。
- `作业验收用例（包含参数说明）.xlsx` 的填写规则也基于 F/T 队列、3 个快充桩、2 个慢充桩、每桩 3 个车位、等候区 10 辆。

当前差距：

- 当前调度方向基本正确，但缺少针对验收用例的明确测试和可解释输出。
- `_find_optimal_pile()` 已计算最短完成时间，但相同完成时间时的稳定规则需要明确。
- `pile_queue_length` 当前容易被理解为总队列长度，验收表要求每桩总容量为 3，需要统一配置语义。
- 普通调度与故障重调度需要明确边界，避免故障优先车辆与普通等候区车辆混合后破坏验收规则。

任务内容：

- 保持普通调度主线为“同类型 FIFO 叫号 + 最短完成时间选桩”，不要在 P0 阶段引入动态优先级评分改变普通队列顺序。
- 明确 `_dispatch_from_waiting_area()` 的车辆选择规则：
  - 快充只从 `fast_waiting` 队首叫号。
  - 慢充只从 `slow_waiting` 队首叫号。
  - 已进入桩队列的车辆不再出现在等候区。
- 明确 `_find_optimal_pile()` 的选桩公式：
  - 等待时间 = 目标桩队列中所有车辆剩余完成时间之和。
  - 自己充电时间 = 请求充电量 / 目标桩功率。
  - 总完成时间 = 等待时间 + 自己充电时间。
  - 选择总完成时间最短的同类型可用桩。
- 对相同总完成时间定义稳定规则，建议按桩编号升序，例如 `F1` 优先于 `F2`。
- 统一每桩队列容量 M：
  - 推荐将 `pile_queue_length` 定义为“每桩总车位数”。
  - 验收配置应为 `pile_queue_length: 3`，表示 1 个正在充电 + 2 个排队。
  - 若决定把 M 定义为“排队车位数”，则代码判断应改成 `len(queue) < 1 + M`，并同步接口文案。
- 补充调度测试：
  - 等候区多辆同类型车按 F/T 排队号顺序进入充电区。
  - 多个同类型桩可用时，选择总完成时间最短的桩。
  - 快充请求不会进入慢充桩，慢充请求不会进入快充桩。
  - 每桩总队列不超过验收配置容量。

验收标准：

- 按验收 Excel 的事件顺序提交请求时，F/T 编号、等候区顺序、桩队列顺序可预测。
- 固定某辆待调度车辆和多个同类型桩队列状态，后端能选择“等待时间 + 自己充电时间”最短的桩。
- 每个桩最多容纳 3 辆车，且第 1 辆为正在充电车辆。
- 快充和慢充保持独立，不互相抢占不同类型桩。
- 普通调度不出现动态优先级评分导致的后到车辆反超，除非进入后续扩展模式。

### BE-P0-05 增加验收快照与当前费用输出

需求依据：

- 验收 Excel 的 C-G 列要求按充电桩填写 `(车号,已充电量,当前费用)`。
- H 列要求填写等候区车辆 `(车号,充电类型,充电量)`。
- 目前系统已有充电量和最终账单，但缺少稳定的“当前时刻全站状态快照”和“充电中当前费用”字段，手工填表容易出错。

任务内容：

- 在 `get_system_status()` 或新增验收接口中返回每个桩的完整队列：
  - `pile_id`
  - `pile_type`
  - `status`
  - `position`
  - `vehicle_id`
  - `queue_number`
  - `requested_kwh`
  - `charged_kwh`
  - `current_fee`
  - `power_fee`
  - `service_fee`
  - `charge_start_time`
- 对正在充电车辆，用 `calculate_fee(charge_start_time, current_vtime, charged_kwh)` 计算当前费用。
- 对桩内排队车辆，返回 `charged_kwh=0`、`current_fee=0`。
- 增加 `GET /api/admin/acceptance/snapshot`，按验收表结构返回：
  - `fast_piles.F1/F2/F3`
  - `slow_piles.T1/T2`
  - `waiting_area`
  - `current_virtual_time`
- 可选：提供脚本读取验收 Excel 的 A/B 列，执行事件并输出 C-H 列结果。

验收标准：

- 管理端可直接拿接口结果填入验收 Excel 的 C-H 列。
- 正在充电车辆的当前费用随虚拟时间推进变化。
- 桩内排队车辆和等候区车辆不会重复出现。
- 任意时刻快充桩、慢充桩、等候区的车辆数量不超过验收容量。

### BE-P0-06 完成改单与取消规则

需求依据：

- 等候区允许修改模式，修改后重新生成排队号并排到新类型队尾。
- 等候区允许修改请求电量，排队号不变。
- 充电区不允许修改，只能取消。

任务内容：

## 四、P2 加分任务

- 动态优先级评分调度：作为后续优化项保留，可引入 `P_i = α * (1 - SOC_i) + β * W_type + γ * ln(t_wait_i + 1)`，但不能影响本次验收默认调度模式。
- 单次多空位调度：多个空位同时出现时，一次调度多辆车，使总完成时长最短。
- 满载批量调度：等待区+充电区满载后进行批量全局优化，不区分快慢充。
- 导出 Excel 文件：在 CSV 基础上增加 `.xlsx` 导出。
- 服务重启状态恢复：从数据库恢复未完成订单和队列。

## 五、后端验收清单

开发完成后至少跑通以下场景：

1. 初始化数据库，确认所有表和 F/T 充电桩存在。
2. 注册车辆并提交快充/慢充请求，生成 F/T 队列号。
3. 多辆车进入同类型等候区，验证按排队号 FIFO 叫号进入充电区。
4. 固定一辆待调度车辆，验证充电桩选择符合最短完成时间。
5. 等候区修改模式，队列号变化并进入新类型队尾；修改电量，队列号不变。
6. 充电区修改请求被拒绝。
7. 正在充电车辆主动结束，生成账单和详单。
8. 设置虚拟时间从 06:00 开始，并按验收表时刻推进，制造跨时段订单，详单分段正确。
9. 充电桩故障，中断车辆结算，故障队列按 A 策略优先调度，故障持续时间结束后自动恢复。
10. 管理员查看日/周/月报表。
11. 导出订单、账单、详单文件。
