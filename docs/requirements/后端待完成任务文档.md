# 后端待完成任务文档

本文档基于《智能充电桩调度计费系统详细需求 20260327(1).docx》、当前代码实现和前期检查结论整理，面向后端开发人员。目标是把系统从“能演示基本流程”推进到“满足详细需求、可稳定验收”的状态。

## 一、当前后端状态

已具备的能力：

- 使用 FastAPI 提供用户、管理员、兼容前端三类接口。
- 已有 `users`、`vehicles`、`charge_orders`、`charging_piles`、`pile_queues`、`pile_status_logs`、`bills`、`bill_details` 等模型。
- 已从 Git 跟踪中移除 `charging.db`，并加入 `.gitignore`，数据库作为本地运行产物处理。
- 服务启动时会创建数据库表，并按 `config.yaml` 种入 F1/F2/F3/T1/T2 充电桩。
- 调度器已有快充/慢充队列、最短完成时间分配、虚拟时间推进、计费、账单/详单生成、故障/恢复处理雏形。
- 管理端已有账单、详单、订单 CSV 导出接口。

主要风险：

- 后端接口存在多套入口：`/api/user/*`、`/api/*`、`/api/admin/*`、兼容接口 `/api/vehicle/*`、`/api/user/charge/*`，需要统一契约。
- 调度运行状态主要存在内存中，数据库中的 `charging_piles`、`pile_queues` 尚未完整反映实时队列。
- 普通调度当前仍是“等候区 FIFO + 选择最短完成时间充电桩”，尚未实现需求规格说明书中的动态优先级评分队列。
- 虚拟时间只支持倍率推进，不支持管理员手动设置当前时间，跨峰/平/谷演示仍不够可控。
- 报表、故障重调度、改单规则需要补充测试闭环。

### BE-P0-04 实现动态优先级评分调度

需求依据：

- 需求规格说明书要求系统采用动态优先级调度算法，而不是简单先到先得。
- 优先级公式为：`P_i = α * (1 - SOC_i) + β * W_type + γ * ln(t_wait_i + 1)`。
- 快充与慢充维护各自独立的优先级队列。
- 入队、完成、主动中止、故障、定时刷新等事件都应触发优先级重算。

当前差距：

- 当前普通调度仍按 `fast_waiting` / `slow_waiting` 列表顺序遍历，车辆选择近似 FIFO。
- 当前只在选择充电桩时实现了“最短完成时间”，没有对等候区车辆计算 `priority_score`。
- `config.yaml` 中还没有 `alpha`、`beta`、`gamma`、`W_type` 等调度权重参数。

任务内容：

- 在 `config.yaml` 中增加调度权重配置，例如：
  - `priority.alpha`
  - `priority.beta`
  - `priority.gamma`
  - `priority.fast_type_weight`
  - `priority.slow_type_weight`
- 在调度器中新增 `calculate_priority(car, current_vtime)`：
  - `SOC_i` 使用 `current_kwh / battery_capacity_kwh`，或由车辆当前电量换算得到。
  - `W_type` 根据快充/慢充配置权重取值。
  - `t_wait_i` 使用当前虚拟时间减去车辆入队时间，单位需统一。
- 将等候区从纯 FIFO 调度改为“同类型队列按优先级分数降序选择车辆”。
- 明确并实现同分规则，例如：
  - 优先级分数高者优先。
  - 分数相同则等待时间长者优先。
  - 仍相同则排队号更早者优先。
- 在以下事件后触发优先级重算：
  - 新请求入队
  - 车辆完成充电释放桩
  - 用户主动结束充电释放桩
  - 用户在等候区修改请求
  - 充电桩故障/恢复
  - 后台定时刷新
- 对返回给前端/管理员端的队列数据增加：
  - `priority_score`
  - `soc`
  - `wait_minutes`
  - `priority_reason`，可选，用于解释排序变化
- 补充单元测试或端到端测试，证明低电量、等待时间长、模式权重会影响调度顺序。

验收标准：

- 构造两个同类型等待车辆，后到但低 SOC 的车辆可以因优先级更高被先调度。
- 等待时间增长后，车辆优先级会随 `ln(t_wait_i + 1)` 增加。
- 修改 `alpha/beta/gamma` 后，同一批等待车辆排序会发生可解释变化。
- 普通调度日志或接口返回能看到优先级分数，证明不是 FIFO。
- 快充和慢充仍保持独立队列，不互相抢同一类型桩。

### BE-P0-05 完成最短完成时间选桩闭环

需求依据：

- 在选出某一辆待调度车辆后，充电桩选择策略为“等待时间 + 自己充电时间”最短。
- 快充号码 F，慢充号码 T，分别进入对应类型桩。

任务内容：

- 确认 `_find_optimal_pile()` 对每个同类型可用桩计算：
  - 队列中所有车剩余充电时长
  - 当前车辆充电时长
  - 两者总和
- 对相同总时长的情况定义稳定规则，例如按桩编号排序。
- 补充测试覆盖：
  - 空桩直接进入
  - 多个桩均有队列时选择总完成时间最短的桩
  - 快充请求不会进入慢充桩，慢充请求不会进入快充桩

验收标准：

- 给定固定队列状态和新请求，调度结果可预测。
- 测试数据能证明不是简单 FIFO 分配到第一个空桩。

### BE-P0-06 完成改单与取消规则

需求依据：

- 等候区允许修改模式，修改后重新生成排队号并排到新类型队尾。
- 等候区允许修改请求电量，排队号不变。
- 充电区不允许修改，只能取消。
- 等候区和充电区均允许取消。

任务内容：

- 暴露正式改单接口，例如 `PUT /api/user/requests/{order_id}`。
- 等候区改单时同步内存队列和数据库订单。

- 取消等待区订单时不生成费用。

验收标准：

- 等候区只修改电量时队列号不变。
- 充电区改单被拒绝。
- 充电中取消后生成账单和详单。

## 五、后端验收清单

开发完成后至少跑通以下场景：

1. 初始化数据库，确认所有表和 F/T 充电桩存在。
2. 注册车辆并提交快充/慢充请求，生成 F/T 队列号。
3. 多辆车进入同类型等候区，验证动态优先级分数会改变调度顺序。
4. 固定一辆待调度车辆，验证充电桩选择符合最短完成时间。
5. 等候区修改模式，队列号变化；修改电量，队列号不变，并触发优先级重算。
6. 充电区修改请求被拒绝。
7. 正在充电车辆主动结束，生成账单和详单。
8. 设置虚拟时间制造跨时段订单，详单分段正确。
9. 充电桩故障，中断车辆结算，后续车辆重新调度。
10. 管理员查看日/周/月报表。
11. 导出订单、账单、详单文件。
