## 2. 调度策略说明

### 2.1 正常调度策略

详细需求中的正常调度不是“全局动态优先级评分”，也不是单纯随机分配，而是两层逻辑：

1. 等候区按充电类型分别排队。
   - 快充请求生成 `F` 前缀排队号。
   - 慢充请求生成 `T` 前缀排队号。
   - 充电区同类型桩有空位时，选取等候区中匹配模式的第一辆车进入充电区。
2. 进入充电区后，选择完成充电所需时长最短的同类型充电桩。
   - 等待时间 = 该桩队列中所有车辆完成充电时间之和。
   - 自己充电时间 = 请求充电量 / 充电桩功率。
   - 目标 = `等待时间 + 自己充电时间` 最短。

当前代码基本采用这一策略：

- `_dispatch_from_waiting_area()` 从 `fast_waiting` / `slow_waiting` 顺序遍历等待队列。
- `_find_optimal_pile()` 按 `pile.remaining_time_hours() + requested_kwh / pile.power` 选择最优桩。
- `submit_request()` 在有同类型充电桩空位时会直接分配到最优桩，无空位时进入等待区。

结论：当前正常调度逻辑与详细需求的主调度策略大体一致。它不是之前任务文档里提到的“动态优先级评分调度”。动态优先级评分更像扩展需求或后续优化项，若本次验收严格依据详细需求 docx 与验收 Excel，应优先保证“同类型 FIFO 叫号 + 最短完成时长选桩”正确。

### 2.2 故障后的 A / B 策略

结合详细需求和验收表说明，可将两种故障重调度策略定义为：

| 策略 | 名称 | 具体内容 | 适用说明 |
| --- | --- | --- | --- |
| A | 优先级调度 | 暂停等候区叫号服务；当其它同类型充电桩队列有空位时，优先调度故障充电桩队列中的车辆；故障队列全部进入充电区后，再恢复等候区叫号。 | 验收 Excel 的注意事项明确写明：充电桩损坏验收时默认采用该策略。 |
| B | 时间顺序调度 | 暂停等候区叫号服务；将其它同类型充电桩中尚未充电车辆与故障桩队列车辆合并，按排队号先后顺序重新调度；调度完毕后恢复等候区叫号。 | 详细需求列出的另一种故障重调度策略；故障恢复时也使用类似时间顺序重排。 |

当前系统状态：

- `fault_pile()` 中，故障发生后对故障桩队列调用 `_priority_dispatch()`，即采用 A 策略。
- `recover_pile()` 中，故障恢复后如果其它同类型桩仍有排队车辆，则调用 `_time_order_dispatch()`，即采用 B 策略思想。

验收建议：

- 本次作业验收应明确采用 A 策略作为充电桩损坏场景默认策略，因为 Excel 说明页已写明“充电桩出现损坏验收时默认采用优先级调度”。
- B 策略可保留为恢复或扩展策略，但测试报告中不要把 B 描述为故障损坏验收的默认策略，避免与验收表冲突。

## 3. 验收 Excel 表格填写指导

### 3.1 Sheet 结构

`需填写sheet` 主要列含义如下：

| 列 | 含义 | 填写要求 |
| --- | --- | --- |
| A | 时刻 | Excel 时间小数。`0.25` 表示 6:00。验收说明中起始时间为早上 6 点。 |
| B | 事件 | 事件四元组 `(事件类型, 对应id, 充电类型, 数值)`。 |
| C | 快充1 | 填写快充 1 桩队列状态。 |
| D | 快充2 | 填写快充 2 桩队列状态。 |
| E | 快充3 | 填写快充 3 桩队列状态。 |
| F | 慢充1 | 填写慢充 1 桩队列状态。 |
| G | 慢充2 | 填写慢充 2 桩队列状态。 |
| H | 等候区 | 填写等候区车辆列表，最大 10 辆。 |

表头提示：

- C-G 列格式为 `(车号,已充电量,当前费用)`。
- H 列格式为 `(车号,充电类型,充电量)`。

### 3.2 事件填写规则

事件四元组格式为：

```text
(事件类型, 对应id, 充电类型, 数值)
```

事件类型：

- `A`：Apply，提交充电请求。
- `B`：Breakdown，充电桩故障。
- `C`：Change，变更充电请求。

对应 id：

- `Vi`：第 i 号电动车。
- `Fj`：第 j 个快充电桩。
- `Tk`：第 k 个慢充电桩。

充电类型：

- `F`：快充。
- `T`：慢充。
- `O`：不变更，或对充电桩故障事件无意义。

数值：

- 当 id 为车辆时，表示请求充电量，单位为度。
- 车辆请求充电量为 `0` 时，代表取消充电。
- 车辆请求充电量为 `-1` 时，代表充电量不变。
- 当 id 为充电桩时，表示故障持续时间，单位为分钟。

### 3.3 充电桩 C-G 列填写方式

每个充电桩单元格建议按队列顺序填写，队列最多 3 个车位：

```text
(V1,10.00,18.00)
(V5,0.00,0.00)
(V8,0.00,0.00)
```

填写原则：

- 第 1 行表示该桩队列第 1 辆车，即正在充电车辆。
- 第 2、3 行表示该桩内排队车辆。
- 正在充电车辆的“已充电量”和“当前费用”应按当前时刻实时计算。
- 桩内排队车辆尚未开始充电，已充电量通常填 `0.00`，当前费用填 `0.00`。
- 空桩可留空，或统一填 `--`，但整张表应保持同一种风格。
- 故障桩在故障时刻应体现故障状态；若表格只允许按车辆格式填写，可在单元格补充 `故障中` 或将车辆清空后在备注中说明。

### 3.4 等候区 H 列填写方式

等候区单元格按等待顺序填写，最多 10 辆：

```text
(V12,T,10)
(V14,F,75)
(V15,F,45)
```

填写原则：

- 只填写尚未进入任何充电桩队列的车辆。
- 已进入某个充电桩队列的车辆应从 H 列移除，写入对应充电桩列。
- 取消充电的车辆应从等候区或充电桩队列中移除。
- 变更充电类型时，若车辆在等候区：
  - 重新生成对应类型排队号。
  - 排到变更后类型队列最后。
  - H 列中车辆的充电类型同步更新。
- 变更充电量但充电类型不变时，排队号不变，H 列中充电量更新。

### 3.5 时间填写规则

验收说明给出的时间规范：

- 起始输入时间为 6:00。
- 比例尺为 1:10。
- 30 分钟真实输入过程对应 300 分钟系统时间，即 5 小时。
- 因此验收事件覆盖约 6:00 至 11:00 的系统时段。
- 计费应按系统虚拟时间跨越峰、平、谷时段计算。

Excel 中的时间是日期小数：

- `0.25` = 6:00。
- `0.253472222222222` = 6:05。
- `0.416666666666667` = 10:00。
- `0.4375` = 10:30。

填写时应以 A 列时刻为系统当前时刻，计算每辆正在充电车辆到该时刻的累计充电量和累计费用。

## 4. 当前系统逻辑对比结果

| 检查项 | 需求 / 验收要求 | 当前实现 | 结论 |
| --- | --- | --- | --- |
| 正常叫号 | 同类型等候区第一辆车被叫入充电区 | `_dispatch_from_waiting_area()` 按等待列表顺序处理 | 基本符合 |
| 选桩策略 | 选择“等待时间 + 自己充电时间”最短的同类型桩 | `_find_optimal_pile()` 已实现该公式 | 基本符合 |
| 快慢充隔离 | 快充车只能进快充桩，慢充车只能进慢充桩 | 按 `charge_type` 筛选同类型桩 | 符合 |
| 排队号 | 快充 F 前缀，慢充 T 前缀，从 1 递增 | `_next_queue_number()` 实现 F/T 独立计数 | 符合 |
| 每桩队列长度 | 每个桩队列长度为 M；验收表为 3 个车位，即 1 充电 + 2 排队 | 当前 `pile_queue_length` 默认 2，代码中 `len(queue) < max_queue_length` 表示总队列容量为 2 | 不符合验收表，应改为总容量 3 或明确 M 的语义 |
| 等候区容量 | 验收表等候区 10 辆 | `waiting_area_size: 10` | 容量值符合，但总容量判断包含桩内车辆，语义需确认 |
| 故障默认策略 | Excel 说明要求损坏验收默认采用优先级调度 | `fault_pile()` 调用 `_priority_dispatch()` | 方向符合 |
| 故障队列优先期间暂停等候区 | 故障队列未清空前，不应调度等候区 | `_priority_dispatch()` 运行期间设置 `numbering_paused=True`，但函数结束后未保留故障队列待处理状态 | 部分符合；若故障队列一次无法全部进入其它桩，剩余车辆会插回等待区，可能被普通等待区车辆混合 |
| 时间顺序调度 | B 策略需合并其它同类型桩未充电车辆和故障队列，按排队号重排 | `_time_order_dispatch()` 可收集其它桩 position>0 和等待区车辆，但不是故障默认入口 | 可作为恢复/扩展逻辑，需补测试 |
| 6:00 起始时间 | 验收表明确开始时间早上 6 点 | `VirtualClock` 使用 `datetime.now()` 作为虚拟起点 | 不符合，需要支持设置虚拟起始时间 |
| 1:10 时间比例 | 验收表要求 30 分钟对应 300 分钟 | `config.yaml` 当前 `virtual_minutes_per_real_second: 1`，即 1 秒真实时间 = 1 分钟虚拟时间，折算为 1:60 | 不符合，应按验收设置为 10 秒虚拟 / 1 秒真实，或明确事件驱动推进 |
| 任意时间事件输入 | 验收表按 A 列指定时刻驱动事件 | 当前系统只有连续虚拟时钟，无“跳转到指定验收时刻并结算状态”的 API | 不符合，难以稳定复现 Excel 用例 |
| 跨时段计费 | 充电过程跨 7:00、10:00 等边界时需分段计费 | `calculate_fee()` 按分钟拆分峰平谷并生成 segments | 基本符合 |
| 计费起止时间 | 以真实充电开始和停止时间计算 | `_assign_to_pile_queue()` 设置 `charge_start_time`，`_finish_charging()` 设置 `charge_end_time` | 基本符合 |
| 费用展示 | 表格需要当前费用，详单需要充电费、服务费、总费用 | 完成后有账单和详单；充电过程中的“当前费用”未形成统一 API 字段 | 不符合表格填写自动化需求 |
| 修改请求 | 等候区可改模式/电量，充电区不可改 | `modify_request()` 仅允许 waiting list 中修改，桩内拒绝 | 基本符合 |
| 取消请求 | 等候区、充电区均允许取消 | `cancel_request()` 对等待、排队、充电中均处理 | 基本符合 |
| 故障持续时间 | B 事件数值代表故障时间，单位分钟 | `fault_pile()` 没有接收故障持续时间，也没有自动恢复定时器 | 不符合 |
| 报表字段 | 日/周/月、桩编号、次数、时长、电量、费用 | `/api/admin/reports` 已按日/周/月聚合 | 基本符合 |
| 导出文件 | 需要详单/账单数据生成 | 已有 `/export/orders`、`/export/bills`、`/export/bill-details` CSV | 基本符合 |

## 5. 主要不符合项与修改方案

### 5.1 队列容量 M 语义不一致

问题：

- 详细需求和验收表说明为“每个充电桩拥有车位 3，一个正在充电，两个排队”。
- 当前 `config.yaml` 中 `pile_queue_length: 2`，代码判断 `len(queue) < max_queue_length`，因此每桩总容量实际为 2，不是 3。

修改方案：

- 将配置语义统一为“每个桩总车位数”，验收时设为 `3`。
- 或将配置语义改为“排队车位数”，代码判断改为 `len(queue) < 1 + pile_queue_length`。
- 前端设置项文案同步改为“每桩总车位数”或“每桩排队车位数”，避免验收解释冲突。

### 5.2 虚拟时间不满足验收表

问题：

- 验收表起始时间固定为 6:00。
- 当前 `VirtualClock` 以 `datetime.now()` 作为虚拟起点。
- 当前比例 `virtual_minutes_per_real_second: 1` 不等于验收要求的 1:10。
- 缺少“跳到 Excel 指定时刻”的接口，测试时难以稳定填写跨 7:00、10:00 的状态。

修改方案：

- 增加虚拟时钟配置：

```yaml
simulation:
  virtual_start_time: "2026-01-01 06:00:00"
  virtual_seconds_per_real_second: 10
```

- `VirtualClock` 支持：
  - `set_time(datetime)`
  - `advance_to(datetime)`
  - `reset(start_time=None)`
- 增加管理员或测试专用 API：
  - `POST /api/admin/clock/set`
  - `POST /api/admin/clock/advance`
  - `GET /api/admin/clock`
- 验收跑表时，按 Excel A 列时刻推进系统时间，再执行 B 列事件，最后导出当前状态。

### 5.3 故障持续时间未实现

问题：

- Excel 说明中 `(B,F1,O,120)` 表示 F1 故障 120 分钟。
- 当前 `fault_pile(pile_id)` 只设置故障状态，不接收故障分钟数，也不会自动恢复。

修改方案：

- 扩展桩控制接口请求体，增加 `duration_minutes`。
- `fault_pile(pile_id, duration_minutes)` 记录 `fault_until`。
- 后台 watcher 或虚拟时间推进后自动触发 `recover_pile()`。
- Excel 驱动测试时，到故障结束时刻应自动恢复该桩，并执行恢复后的时间顺序调度。

### 5.4 故障 A 策略剩余车辆保护不足

问题：

- A 策略要求故障队列全部调度完毕前，暂停等候区叫号。
- 当前 `_priority_dispatch()` 如果其它同类型桩暂时没有空位，会把剩余故障车辆插回普通等待列表前端。
- 这种实现虽然大概率让故障车辆排在前面，但没有单独的“故障优先队列”状态；后续新增或变更请求可能造成规则解释不清。

修改方案：

- 新增 `fast_fault_waiting` / `slow_fault_waiting`，或给 waiting item 增加 `source='fault_queue'`。
- 普通调度前先清空故障优先队列。
- 只有故障优先队列为空时，才允许调度普通等候区。
- 管理端展示中单独标识“故障重调度车辆”。

### 5.5 当前费用字段不足，无法自动填表

问题：

- 验收表 C-G 列要求 `(车号,已充电量,当前费用)`。
- 当前系统在充电中可计算 `charged_kwh`，但接口没有稳定返回“截至当前时刻的当前费用”。

修改方案：

- 在 `get_system_status()` 的 `queue_items` 中增加：
  - `current_fee`
  - `power_fee`
  - `service_fee`
  - `current_bill_segments`
  - `charge_start_time`
- 对正在充电车辆调用 `calculate_fee(charge_start_time, current_time, charged_kwh)`。
- 对桩内排队车辆返回 `charged_kwh=0`、`current_fee=0`。

### 5.6 缺少验收表自动导出能力

问题：

- 目前已有订单、账单、详单 CSV 导出，但没有按 Excel 验收表格式导出“某时刻全站状态”。
- 手动填表容易出错，特别是跨时段费用和故障重调度。

修改方案：

- 增加测试辅助接口：
  - `GET /api/admin/acceptance/snapshot`
  - 返回 C-G 各桩队列、H 等候区、当前时间、当前事件处理结果。
- 增加脚本：
  - 读取 `作业验收用例（包含参数说明）.xlsx`
  - 按 A/B 列事件驱动系统
  - 自动写回 C-H 列
  - 输出 `作业验收用例_系统结果.xlsx`

### 5.7 计费分段基本正确，但需补边界测试

当前 `calculate_fee()` 按分钟遍历时间区间，能处理跨 7:00、10:00、15:00、18:00、21:00、23:00 的费用分段。

风险：

- 按分钟计费可能对秒级边界产生舍入误差。
- Excel 用例按 5 分钟或更大粒度推进，通常可接受，但仍需固定验收精度。

修改方案：

- 明确验收口径：金额保留 2 位，电量保留 2 位或 4 位。
- 增加单元测试：
  - 6:50-7:10，跨谷/平。
  - 9:50-10:10，跨平/峰。
  - 22:50-23:10，跨平/谷。

## 6. 建议验收执行方案

1. 固定验收配置：
   - 快充桩：3。
   - 慢充桩：2。
   - 每桩总车位：3。
   - 等候区容量：10。
   - 快充功率：30 度/小时。
   - 慢充功率：10 度/小时。
   - 虚拟起始时间：6:00。
   - 时间比例：1:10。
2. 明确故障策略：
   - 损坏验收默认采用 A 策略，即优先级调度。
   - B 策略作为恢复或补充验证，不作为 Excel 损坏场景默认结论。
3. 先补齐测试辅助能力：
   - 虚拟时间设置/推进。
   - 故障持续时间和自动恢复。
   - 当前费用实时返回。
   - 验收快照导出。
4. 再按 Excel 顺序执行事件：
   - 每到一个 A 列时刻，先推进虚拟时间。
   - 结算当前所有正在充电车辆的电量和费用。
   - 执行 B 列事件。
   - 触发调度。
   - 记录 C-H 列状态。
5. 最后核对：
   - 每辆车只在一个区域出现。
   - 桩队列不超过 3 个车位。
   - 等候区不超过 10 辆。
   - 快慢充不混入错误类型桩。
   - 故障期间故障桩不再接车。
   - 故障队列清空前普通等候区不抢占。
   - 跨 10:00 后费用进入峰时电价。

## 7. 总体结论

当前系统已经具备验收所需的核心雏形：

- 快慢充队列独立。
- 正常选桩采用最短完成时长。
- 车辆请求、修改、取消、结束充电已有后端逻辑。
- 分时计费、账单、详单、报表和 CSV 导出已有基础实现。
- 故障场景默认方向接近验收要求的优先级调度。

但若要严格通过 `作业验收用例（包含参数说明）.xlsx`，还需要优先修复以下问题：

1. 统一每桩队列容量 M 的语义，验收应支持每桩 3 个车位。
2. 支持固定从 6:00 开始的虚拟时间和 1:10 时间比例。
3. 支持按 Excel 指定时刻推进系统，而不是依赖真实时间自然流逝。
4. 故障事件必须支持持续时间和自动恢复。
5. 故障优先队列需要单独保护，确保 A 策略不会被普通等候区插队破坏。
6. 管理端或测试接口需要返回“当前费用”，否则无法可靠填写 C-G 列。
7. 建议新增验收表自动导出脚本，减少人工填表误差。
